amazon s3 (simple storage service)

s3 bucket 생성

https://s3.console.aws.amazon.com/s3 접속 후 버킷만들기

버킷의 이름은 다른 버킷의 이름과 중복되면 안됨. aws 리전의 경우 서울리전으로 선택하는 것이 일반적이지만, cloudfront 적용 전후의 속도비교를 위해 거리가 먼 버지니아 리전으로 선택

나머지 부분은 기본값으로 선택함

object(file) 업로드

만든 버킷으로 들어가 업로드 버튼을 클릭

업로드 화면에서 파일탐색기의 파일을 드래그+드롭하거나 파일추가, 폴더추가 버튼으로 원하는 파일들을 업로드할 수 있음

정적 웹사이트 호스팅 기능 활성화

버킷 페이지로 돌아가 속성탭의 가장 하단의 정적 웹사이트 호스팅 카드에서 편집버튼 클릭

정적 웹사이트 호스팅을 활성화하면 호스팅 유형~ 리디렉션 규칙까지 관련된 속성들이 표시됨.

객체에 대한 요청 리디렉션은 들어오는 요청을 다른 버킷이나 도메인으로 리디렉션 하는 속성

인덱스 문서는 버킷에 들어오는 요청에 해한 default page를 설정하는 속성이며 이전에 업로드했던 mycar.html을 설정함

설정 후 이전 페이지로 돌아가면 설정한 정적 웹 사이트에 대한 주소가 표시됨. 그러나 아직 외부 사용자에 대한 접근 권한이 부여되지 않았기 때문에 해당 사이트에 접속해도 403 Forbidden을 응답함.

bucket, object에 대한 access policy 설정

퍼블릭 엑세스를 허용하기 위해 버킷 페이지에서 권한탭의 퍼블릭 엑세스 차단메뉴를 편집

퍼블릭 액세스 차단 편집창에서 모든 퍼블릭 엑세스 차단의 체크를 해제시켜주고 저장

퍼블릭 액세스가 허용되었으면 버킷에 대한 object(파일)에 대한 접근 또한 허용해주어야 정상적으로 웹페이지를 읽어올 수 있을 것임. 따라서 퍼블릭 액세스 차단 메뉴 아래 버킷 정책 메뉴에서 버킷 정책을 편집해주어야 함.

버킷 정책 메뉴에서 편집을 누른 뒤 버킷 정책 편집 페이지에서 왼쪽 상단 정책 생성기 클릭

policy generator에서 위 사진대로 속성을 입력함. (actions는 getObject 1개 설정)

arn의 경우 이전의 버킷 정책 편집 페이지 상단부분에 있음

policy generator에서 올바른 arn을 입력해도 arn이 invalid한 값이라고 오류가 발생했는데 자바스크립트 콘솔에서 String의 match 함수를 수정해서 넘어감.
(참조: https://stackoverflow.com/questions/72756697/resource-field-is-not-valid-you-must-enter-a-valid-arn)

이후 generate policy로 나온 policy json을 그대로 복사해 버킷 정책 편집 페이지에 붙여넣기 함.

웹사이트 접속 및 작동 여부 확인

다시 속성탭의 정적 웹사이트 호스팅의 웹사이트 주소로 접속하면 이전과 달리 접속이 가능함.

amazon cloudfront

cloudfront distribution 생성

https://us-east-1.console.aws.amazon.com/cloudfront 접속 후 배포 생성 클릭

원본도메인으로 s3를 선택함

원본경로는 cloudfront가 s3의 특정 내부 디렉토리에서 콘텐츠를 요청하는 경우 해당하는 세부 경로를 요청하는 속성을 의미함.

s3 버킷 엑세스는 OAI를 통해 cloudfront의 origin이 되는 s3에 대한 access를 컨트롤 할 것인지에 대한 설정을 의미함.

자동으로 객체 압축 속성은 cloudfront가 origin에서 가져온 object의 전송속도, 효율을 위해 자동으로 압축하는 기능임

cloudfront는 edge location에서 cache hit가 발생할 때 origin server의 개입 없이 edge location에서 콘텐츠를 가져오기 때문에 origin server의 부하를 줄이고 사용자의 delay time을 줄일 수 있음. 해당 내용에 대한 정책들은 캐시 키 및 원본 요청 속성에서 설정할 수 있음.

특정 header값에 따라서 캐시 정책을 바꿔야 하는 경우에는 legacy cache settings를 선택해 커스터마이징이 가능함.

가격 분류는 어느 지리적 위치에 cloudfront를 적용할 것인지를 선택하는 속성으로 많은 위치에 적용시킬수록 더 좋은 효율을 얻을 수 있지만 가격이 비싸진다.

기본값 루트 객체는 루트로 요청할 때 어떤 object를 반환할 것인지 설정하는 속성임. 이번 경우에는 s3의 정적 웹 호스팅과 cloudfront의 속도비교를 위해 s3의 정적 웹 호스팅의 경우와 같은 mycar.html로 설정.

브라우저에서 cloudfront distribution 작동 확인

배포 후 cloudfront 콘솔에서 도메인 주소와 상태를 확인할 수 있음

접속하면 s3의 mycar.html과 동일한 화면을 확인할 수 있으며 개발자도구로 응답의 header의 x-cache가 Hit from cloudfront임을 통해 cloudfront를 통해 load가 되었음을 확인할 수 있다.

cloudfront 적용 유무에 따른 콘텐츠 로드 속도 확인

s3로 호스팅된 웹페이지에서 mycar.html을 로드하는데 평균적으로 3초 가량 걸리는 반면, cloudfront로 로딩된 웹페이지에서는 1.5초 가량 소요되었으며 cloudfront를 적용한 방법이 더 빠른 콘텐츠 로드 속도를 보임.
